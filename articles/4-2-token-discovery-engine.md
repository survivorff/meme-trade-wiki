# Token 发现引擎：如何实时捕获新币

> 每天数万个新 Token 被创建，平台需要在毫秒级时间内发现、分析、展示它们。

## 表面认知

"监听一下链上事件不就行了？" 监听只是第一步，后面的处理链路才是难点。

## 真实情况

### 数据源

**1. Pump.fun Token 创建事件**
- 监听 Pump.fun 程序的 `create` 指令
- 每个新 Token 创建时会触发链上事件
- 包含 Token 地址、创建者、名称、图片等元数据

**2. DEX 流动性池创建事件**
- 监听 Raydium、PumpSwap、Meteora 等 DEX 的池创建事件
- Token 毕业后会在 DEX 上创建流动性池
- 这是另一个重要的发现时机

**3. 其他发射台**
- LetsBonk、Moonshot 等平台的创建事件
- 每个发射台的合约接口不同，需要分别适配

### 技术实现

**方式一：WebSocket 订阅**
```
连接 Solana RPC WebSocket
    → 订阅相关程序的日志（logsSubscribe）
    → 实时接收新区块中的相关事件
    → 解析事件数据
```
- 优点：实时性好，延迟低
- 缺点：连接不稳定，可能丢消息

**方式二：Geyser Plugin / gRPC**
```
连接支持 Geyser 的 RPC 节点
    → 通过 gRPC 流式接收账户变更
    → 过滤相关程序的账户更新
    → 解析数据
```
- 优点：更可靠，数据更完整
- 缺点：需要支持 Geyser 的节点，成本更高

**方式三：轮询（Polling）**
```
定时查询最新区块
    → 解析区块中的所有交易
    → 过滤相关程序的交易
    → 提取事件数据
```
- 优点：实现简单，不会丢消息
- 缺点：延迟较高（取决于轮询间隔）

**实际方案：多种方式组合**
- 主路径：WebSocket/Geyser 实时接收
- 备用路径：轮询补漏
- 两者交叉验证，确保不遗漏

### 发现后的处理流程

```
新 Token 事件到达
    │
    ▼
基础信息提取
    ├── Token 地址、名称、符号、图片
    ├── 创建者地址
    ├── 初始参数（供应量、Bonding Curve 参数）
    └── 创建时间
    │
    ▼
安全快速检查（毫秒级）
    ├── 创建者是否有 Rug Pull 历史？
    ├── Mint/Freeze 权限状态？
    └── 是否是已知的恶意模式？
    │
    ▼
入库 + 推送
    ├── 写入数据库
    ├── 推送到前端（WebSocket）
    └── 触发后续的深度分析任务
    │
    ▼
持续监控
    ├── 价格变化
    ├── 交易量
    ├── 持仓分布变化
    └── 社交媒体热度（如果有集成）
```

### 过滤策略

每天数万个新 Token，不可能全部展示。平台需要过滤策略：

**基础过滤（自动）：**
- 排除已知恶意创建者的 Token
- 排除明显的蜜罐合约
- 排除没有任何交易活动的 Token

**排序因素：**
- 交易量和交易笔数
- 独立买家数量
- 价格变化幅度
- 创建者历史信誉
- 社交媒体提及量

### 性能挑战

- **数据量**：高峰期每秒可能有数十个新 Token 创建
- **处理延迟**：从事件到展示需要控制在 1-3 秒内
- **存储**：1300 万+ Token 的历史数据
- **并发查询**：大量用户同时刷新新币列表

## 经验教训

1. **可靠性比速度更重要** —— 漏掉一个热门 Token 比晚发现 1 秒更严重
2. **多数据源交叉验证** —— 不要依赖单一数据源
3. **过滤策略需要持续调优** —— 市场变化快，过滤规则要跟上
4. **缓存是性能的关键** —— 热门数据要缓存，减少链上查询
