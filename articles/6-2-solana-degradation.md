# 当 Solana 宕机：网络降级下的平台生存策略

> Solana 不是永远在线的。网络降级时，平台怎么保护用户和自己？

## 表面认知

"Solana 宕机了就等着呗。" 对用户来说可以等，但对平台来说，宕机期间的每一秒都可能造成损失。

## 真实情况

### Solana 网络降级的类型

**1. TPS 下降**
- 正常 TPS：3000-4000
- 降级时可能降到 500-1000
- 交易确认变慢，失败率上升

**2. 区块生产停滞**
- 极端情况下，区块生产可能暂停
- 所有交易都无法确认
- 历史上发生过多次

**3. RPC 节点过载**
- 网络拥堵时 RPC 请求量暴增
- 节点可能限流或宕机
- 平台无法查询数据或提交交易

### 对交易平台的影响

| 影响 | 严重程度 | 说明 |
|------|---------|------|
| 交易无法执行 | 🔴 | 用户无法买卖 |
| 止损无法触发 | 🔴 | 用户可能遭受额外损失 |
| 价格数据延迟 | 🟡 | 显示的价格可能不准确 |
| 新币发现延迟 | 🟡 | 错过新 Token |
| 跟单延迟 | 🟡 | 跟单价格偏差增大 |

### 降级应对策略

**1. 检测与告警**
```
监控指标：
  - Solana TPS（实时）
  - 区块生产间隔
  - RPC 响应时间
  - 交易确认延迟
  - Landing Rate

告警阈值：
  - TPS < 1000 → 黄色告警
  - 区块间隔 > 2s → 橙色告警
  - 区块停产 > 10s → 红色告警
```

**2. 自动降级**
- 检测到网络降级时，自动调整平台行为：
  - 提高 Priority Fee 建议值
  - 增加交易重试次数
  - 暂停非关键功能（如数据分析）
  - 在 UI 上提示用户网络状况

**3. 止损保护**
- 网络降级时，止损交易可能无法执行
- 平台应该：
  - 持续重试止损交易
  - 使用最高优先级提交
  - 通知用户止损可能延迟
  - 记录所有未执行的止损，网络恢复后立即执行

**4. 用户沟通**
- 实时状态页面展示网络状况
- 推送通知告知用户
- 透明说明影响范围和预计恢复时间

### 事后复盘

每次网络降级事件后：
1. 统计影响范围（多少用户受影响、多少交易失败）
2. 分析平台的应对是否及时
3. 识别改进点
4. 更新降级预案

## 经验教训

1. **为最坏情况做准备** —— Solana 网络降级不是"如果"而是"什么时候"
2. **止损保护是最高优先级** —— 网络降级时，保护用户资金比任何功能都重要
3. **透明沟通建立信任** —— 用户可以接受问题，但不能接受被蒙在鼓里
4. **自动化降级比人工响应更可靠** —— 凌晨 3 点的网络降级不会等你起床
