# 从 100 到 10 万用户：实时交易平台的扩容之路

> 热门 Token 发射时，流量可能在几分钟内暴增 100 倍。平台要么扛住，要么崩溃。

## 表面认知

"加服务器就行了。" 加服务器能解决一部分问题，但瓶颈往往不在服务器数量。

## 真实情况

### 扩容的三个阶段

**阶段一：100-1000 用户（单体架构）**
- 一个后端服务处理所有逻辑
- 单数据库实例
- 够用，但没有冗余

**阶段二：1000-10000 用户（服务拆分）**
- 交易引擎、数据服务、用户服务分离
- 数据库读写分离
- 引入消息队列处理异步任务
- WebSocket 服务独立部署

**阶段三：10000-100000 用户（分布式架构）**
- 各服务水平扩展
- 数据库分片
- 多级缓存
- 全球多区域部署

### 常见瓶颈

| 瓶颈 | 表现 | 解决方案 |
|------|------|---------|
| RPC 调用量 | 链上查询超限 | 缓存 + 多 RPC + 批量查询 |
| WebSocket 连接数 | 推送延迟增大 | 水平扩展 + 连接分片 |
| 数据库写入 | 交易记录写入慢 | 写缓冲 + 异步写入 + 分库 |
| 交易提交并发 | 交易排队 | 多通道并行提交 |
| 行情计算 | K 线更新延迟 | 增量计算 + 按需计算 |

### 热点问题

Meme 交易平台有一个特殊的扩容挑战：**热点 Token**。

当一个 Token 突然爆火时：
- 所有用户同时查看同一个 Token 的数据
- 大量用户同时交易同一个 Token
- 这个 Token 的数据更新频率暴增

这是典型的"热点"问题，普通的水平扩展解决不了。

**应对策略：**
- 热点 Token 的数据单独缓存，更高的刷新频率
- 交易请求排队 + 限流，防止系统过载
- 前端做乐观更新，减少后端压力

### 成本与性能的平衡

扩容不是免费的：
- 更多服务器 = 更高的云服务费用
- 更多 RPC 调用 = 更高的节点费用
- 更复杂的架构 = 更高的运维成本

关键是找到**成本效率的甜蜜点**：
- 不要为峰值流量常备资源（太贵）
- 使用弹性扩容，按需增减
- 识别哪些组件需要常备冗余（交易引擎），哪些可以弹性（数据分析）

## 经验教训

1. **提前规划，但不要过度设计** —— 在需要之前就准备好扩容方案
2. **瓶颈通常不在你以为的地方** —— 用数据说话，不要猜
3. **热点问题需要专门处理** —— 通用的扩容方案解决不了热点
4. **成本意识很重要** —— 尤其在熊市，收入下降但基础设施成本不会
