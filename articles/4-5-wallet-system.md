# 钱包系统设计：托管、非托管与混合方案

> 钱包是用户资金的容器，也是平台最敏感的组件。设计不好，轻则体验差，重则资金被盗。

## 表面认知

"让用户连接 Phantom 钱包不就行了？" 对于简单的 DApp 是这样，但对于需要自动交易的 Meme 平台，远远不够。

## 真实情况

### 三种钱包模式

**1. 完全非托管**
- 用户通过 Phantom/Backpack 等钱包连接
- 每笔交易需要用户手动签名确认
- 平台完全不接触私钥

优点：最安全，用户完全控制资金
缺点：无法实现自动交易（止盈止损、跟单），每笔交易都要手动确认，速度慢

**2. 完全托管**
- 平台为用户生成钱包，私钥存储在平台服务器
- 用户充值到平台钱包，交易由平台代为执行
- 类似中心化交易所的模式

优点：速度最快，可以实现所有自动化功能
缺点：平台持有私钥，安全风险极高（被黑客攻击、平台跑路）

**3. 混合模式（主流方案）**
- 用户可以选择连接外部钱包（非托管）
- 也可以在平台内创建托管钱包（用于自动交易）
- 用户自行决定在托管钱包中放多少资金

优点：兼顾安全和功能
缺点：实现复杂，用户需要理解两种模式的区别

### 托管钱包的安全架构

如果平台提供托管钱包，密钥管理是核心挑战：

```
用户创建钱包
    │
    ▼
生成密钥对
    │
    ▼
私钥加密存储
    ├── 使用 HSM（硬件安全模块）或 KMS（密钥管理服务）
    ├── 私钥永远不以明文形式存在于内存中
    └── 加密密钥和数据密钥分离
    │
    ▼
签名服务
    ├── 独立的签名服务，与业务逻辑隔离
    ├── 签名请求需要经过权限验证
    ├── 所有签名操作记录审计日志
    └── 限制单笔和每日签名金额
```

**关键安全措施：**
- 私钥加密存储，使用 AWS KMS / GCP Cloud KMS 等
- 签名服务独立部署，网络隔离
- 多层权限控制（谁能触发签名、签名金额限制）
- 完整的审计日志
- 定期安全审计

### 资金安全的其他考量

**热钱包 vs 冷钱包：**
- 热钱包：在线，用于日常交易，金额应该尽量少
- 冷钱包：离线，存储大额资金，需要多签才能动用
- 平台应该定期将热钱包中的多余资金转移到冷钱包

**多签（Multi-sig）：**
- 平台的运营资金应该使用多签钱包
- 大额转账需要多个管理员确认
- 防止单点故障（一个人的密钥泄露不会导致资金损失）

### 用户体验设计

| 操作 | 非托管钱包 | 托管钱包 |
|------|-----------|---------|
| 买入 | 弹出钱包确认 → 签名 → 等待 | 点击买入 → 直接执行 |
| 止盈止损 | 无法实现（或需要预授权） | 自动执行 |
| 跟单 | 无法实现 | 自动执行 |
| 提现 | 不需要（资金在自己钱包） | 需要提现到外部钱包 |

## 经验教训

1. **安全是底线** —— 一次安全事故可以毁掉整个平台
2. **给用户选择权** —— 不要强制用户使用托管钱包
3. **最小化托管资金** —— 提醒用户只在托管钱包中放需要交易的金额
4. **安全审计不能省** —— 定期请第三方审计密钥管理系统
