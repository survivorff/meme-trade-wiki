# 系统架构总览：一个 Meme 交易平台的组成

> 从外面看是一个网页，从里面看是一个由十几个子系统组成的实时数据处理平台。

## 表面认知

"不就是一个前端页面加一个后端 API 吗？" 如果只是这样，就不需要一个工程团队了。

## 真实情况

### 核心组件

一个完整的 Meme 交易平台至少包含以下子系统：

```
┌─────────────────────────────────────────────────────────┐
│                      用户层                              │
│  Web 前端 · Telegram Bot · Mobile App · API             │
├─────────────────────────────────────────────────────────┤
│                      业务层                              │
│  交易引擎 · 跟单系统 · 自动交易 · Token 安全评分         │
├─────────────────────────────────────────────────────────┤
│                      数据层                              │
│  Token 发现 · 行情数据 · 钱包追踪 · 链上索引             │
├─────────────────────────────────────────────────────────┤
│                      基础设施层                           │
│  RPC 管理 · Jito 集成 · 交易广播 · 钱包服务              │
├─────────────────────────────────────────────────────────┤
│                      链上层                              │
│  Solana · Raydium · Jupiter · Pump.fun · PumpSwap       │
└─────────────────────────────────────────────────────────┘
```

### 各子系统职责

**1. Token 发现引擎**
- 监听链上事件（新 Token 创建、新流动性池）
- 实时过滤和评分
- 推送到前端展示
- 延迟要求：秒级

**2. 行情数据管线**
- 从链上原始交易数据计算价格
- 生成 K 线（OHLCV）数据
- 通过 WebSocket 实时推送给前端
- 延迟要求：秒级，数据量巨大

**3. 交易引擎**
- 接收用户的交易请求
- 构建链上交易（选择 DEX、计算路由、设置参数）
- 通过 Jito Bundle 或直接提交
- 监控交易状态，处理失败重试
- 延迟要求：毫秒级

**4. 跟单系统**
- 监控目标钱包的链上活动
- 检测到交易后快速构建跟单交易
- 管理跟单策略（金额、滑点、过滤条件）
- 延迟要求：秒级

**5. 自动交易系统**
- 持续监控用户设置的止盈止损条件
- 条件触发时自动执行交易
- 管理大量并发的监控任务
- 可靠性要求：极高（错过止损可能导致用户重大损失）

**6. Token 安全评分**
- 分析合约权限（Mint、Freeze、LP 状态）
- 分析持仓分布
- 检查创建者历史
- 实时更新评分

**7. 钱包服务**
- 用户钱包管理（托管或非托管）
- 密钥存储和签名
- 余额查询和资产管理

**8. RPC 管理**
- 多 RPC 节点负载均衡
- 故障检测和自动切换
- 不同操作使用不同的 RPC（读操作 vs 写操作）

### 技术选型考量

| 组件 | 常见选择 | 考量因素 |
|------|---------|---------|
| 后端语言 | Go / Rust / TypeScript | 性能 vs 开发效率 |
| 数据库 | PostgreSQL + Redis | 持久化 + 高速缓存 |
| 消息队列 | Kafka / Redis Streams | 高吞吐事件处理 |
| WebSocket | 自建 / Socket.io | 实时数据推送 |
| 链上交互 | Solana Web3.js / Anchor | SDK 成熟度 |
| RPC | Helius / Triton / 自建 | 延迟 + 可靠性 + 成本 |
| 部署 | AWS / GCP / 裸金属 | 延迟敏感场景可能需要裸金属 |

### 数据流示例：一笔买入交易

```
用户点击"买入"
    │
    ▼
前端发送请求到后端 API
    │
    ▼
交易引擎接收请求
    ├── 查询 Token 信息和当前价格
    ├── 选择最优 DEX 和路由
    ├── 计算交易参数（金额、滑点、Compute Unit）
    └── 构建 Solana Transaction
    │
    ▼
签名
    ├── 托管钱包：后端直接签名
    └── 非托管钱包：发回前端让用户钱包签名
    │
    ▼
交易提交
    ├── 通过 Jito Bundle 提交（需要 MEV 保护时）
    └── 直接通过 RPC 提交（普通交易）
    │
    ▼
交易状态监控
    ├── 轮询交易确认状态
    ├── 超时未确认 → 重试或放弃
    └── 确认成功 → 更新用户余额和持仓
    │
    ▼
通知用户交易结果
```

### 最大的技术挑战

1. **实时性**：所有数据都需要实时处理和推送
2. **可靠性**：交易不能丢，止损不能漏
3. **并发量**：热门 Token 发射时，请求量可能瞬间暴增 10-100 倍
4. **链上不确定性**：交易可能失败、网络可能拥堵、RPC 可能宕机
5. **成本控制**：RPC 调用量巨大，成本是主要支出项

## 经验教训

1. **从简单开始，逐步迭代** —— 不要一开始就追求完美架构
2. **链上交互是最大的不确定性来源** —— 要为各种失败场景做好准备
3. **监控比功能更重要** —— 你需要知道系统的每个环节是否正常
4. **RPC 是命脉** —— 节点挂了，整个平台就瘫了
