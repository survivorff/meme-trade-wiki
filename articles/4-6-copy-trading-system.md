# 跟单系统：从钱包监控到自动执行的全链路

> 跟单看起来简单 —— 别人买什么我买什么。但从工程角度看，这是平台中最复杂的子系统之一。

## 表面认知

"监控一个钱包地址，它买了我也买。" 原理没错，但魔鬼在细节里。

## 真实情况

### 跟单系统架构

```
钱包监控层
    ├── 监控数千个目标钱包的链上活动
    ├── 实时解析每笔交易的含义
    └── 识别出"买入 Token X"这样的有效信号
    │
    ▼
信号处理层
    ├── 过滤：这笔交易是否值得跟？
    ├── 匹配：哪些用户在跟踪这个钱包？
    └── 策略：每个用户的跟单参数是什么？
    │
    ▼
交易执行层
    ├── 为每个跟单用户构建交易
    ├── 批量提交（可能同时有数百个跟单交易）
    └── 监控执行结果
```

### 钱包监控的技术挑战

**挑战一：交易解析**
- 链上交易是原始的指令数据，不是"买入 Token X"这样的语义
- 需要解析 DEX 的 Swap 指令，理解交易的含义
- 不同 DEX 的指令格式不同（Raydium vs Jupiter vs PumpSwap）
- 有些交易很复杂（多跳路由、闪电贷）

**挑战二：监控规模**
- 平台可能需要监控数千到数万个钱包
- 每个钱包的每笔交易都需要实时检测
- 需要高效的事件过滤机制

**挑战三：速度**
- 跟单的价值在于"快"
- 从目标钱包交易确认到跟单交易提交，延迟越短越好
- 但太快可能导致误判（交易还没最终确认就跟了）

### 跟单策略参数

用户通常可以配置：

| 参数 | 说明 | 典型值 |
|------|------|--------|
| 跟单金额 | 每次跟单投入多少 | 固定金额 或 按比例 |
| 最大滑点 | 可接受的最大滑点 | 5%-20% |
| 是否跟卖 | 目标钱包卖出时是否也卖 | 是/否 |
| Token 过滤 | 只跟特定类型的 Token | 按市值、流动性等 |
| 单 Token 上限 | 同一个 Token 最多跟多少次 | 1-3 次 |
| 每日上限 | 每天最多跟多少笔 | 10-50 笔 |

### 跟单的结构性问题（再次强调）

从系统设计角度看，跟单有几个无法完全解决的问题：

**1. 延迟价差**
- 目标钱包买入价格：$0.001
- 跟单交易执行价格：$0.0012-$0.0015（延迟 + 跟单买压推高价格）
- 这个价差是跟单者的结构性劣势

**2. 流动性冲击**
- 如果 100 个人同时跟单同一个钱包买入同一个 Token
- 这 100 笔买入会显著推高价格
- 后执行的跟单者买入价格更高

**3. 反向利用**
- 被跟踪的钱包知道有人在跟单
- 可以故意买入 → 等跟单者推高价格 → 卖出获利
- 这是一种合法但不道德的操作

### 平台的应对措施

- **延迟随机化**：不是所有跟单交易同时提交，加入随机延迟
- **总量限制**：限制同一个 Token 的跟单总量
- **异常检测**：检测目标钱包是否有"养号"行为
- **透明度**：展示跟单的历史胜率和平均延迟

## 经验教训

1. **跟单系统的复杂度远超预期** —— 交易解析、并发处理、策略管理都是难点
2. **延迟是跟单的天敌** —— 每减少 100ms 延迟都有价值
3. **要保护跟单者** —— 不能让跟单变成被收割的工具
4. **数据透明是信任的基础** —— 展示真实的跟单胜率和成本
