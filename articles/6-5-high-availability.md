# 高可用与容灾：交易平台不能挂

> 用户可以接受交易慢，但不能接受平台挂了。尤其是当他们有未平仓的持仓时。

## 表面认知

"部署两台服务器做个备份就行了。" 高可用远不止备份这么简单。

## 真实情况

### 可用性目标

| 级别 | 年停机时间 | 适用场景 |
|------|-----------|---------|
| 99% | 3.65 天 | 个人项目 |
| 99.9% | 8.76 小时 | 一般商业应用 |
| 99.95% | 4.38 小时 | 交易平台最低要求 |
| 99.99% | 52.6 分钟 | 理想目标 |

### 关键组件的高可用设计

**1. API 服务**
- 多实例部署 + 负载均衡
- 健康检查 + 自动摘除故障实例
- 无状态设计，任何实例都能处理任何请求

**2. 数据库**
- 主从复制，读写分离
- 自动故障转移（主库挂了从库自动接管）
- 定期备份 + 异地备份

**3. 缓存（Redis）**
- Redis Cluster 或 Sentinel 模式
- 缓存穿透/雪崩防护
- 缓存失效时的降级策略

**4. 消息队列**
- 集群部署，多副本
- 消息持久化
- 消费者组 + 自动重平衡

**5. WebSocket 服务**
- 多实例 + 会话亲和性
- 断线自动重连
- 消息补发机制

### 容灾场景

| 场景 | 影响 | 应对 |
|------|------|------|
| 单台服务器宕机 | 部分请求失败 | 负载均衡自动切换 |
| 数据库主库宕机 | 写操作中断 | 自动故障转移到从库 |
| RPC 服务商故障 | 链上操作中断 | 切换到备用 RPC |
| 整个可用区故障 | 全部服务中断 | 跨可用区部署 + 自动切换 |
| 云服务商故障 | 全部服务中断 | 多云部署（成本很高） |

### 监控与告警

**必须监控的指标：**
- 服务健康状态（每个实例）
- API 响应时间（P50/P95/P99）
- 错误率
- 数据库连接数和查询延迟
- 缓存命中率
- 消息队列积压量
- RPC 可用性和延迟

**告警分级：**
- P0（立即响应）：服务不可用、资金安全问题
- P1（15 分钟内响应）：核心功能降级
- P2（1 小时内响应）：非核心功能异常
- P3（下个工作日）：性能下降、非紧急问题

### 应急预案

每个关键故障场景都应该有预案：
1. 故障检测方式
2. 自动化应对措施
3. 人工介入的触发条件
4. 恢复步骤
5. 事后复盘流程

## 经验教训

1. **高可用是设计出来的，不是加服务器加出来的**
2. **定期做故障演练** —— 不演练的预案等于没有预案
3. **监控覆盖率要高** —— 你不知道的故障是最危险的
4. **自动化恢复优于人工恢复** —— 凌晨的故障不会等你
