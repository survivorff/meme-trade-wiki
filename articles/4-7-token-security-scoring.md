# Token 安全评分系统：自动化风险检测

> 每天数万个新 Token，人工审核不可能。自动化安全评分是平台保护用户的第一道防线。

## 表面认知

"检查一下合约权限不就行了？" 合约权限只是冰山一角。

## 真实情况

### 评分维度

一个完整的 Token 安全评分系统需要检查多个维度：

**1. 合约层面**

| 检查项 | 权重 | 安全 | 危险 |
|--------|------|------|------|
| Mint Authority | 高 | 已撤销 | 未撤销（可无限增发） |
| Freeze Authority | 高 | 已撤销 | 未撤销（可冻结账户） |
| LP Token 状态 | 高 | 已销毁 | 未锁定（可撤走流动性） |
| 合约是否可升级 | 中 | 不可升级 | 可升级（逻辑可被修改） |
| 是否有 Blacklist | 中 | 无 | 有（可阻止卖出） |
| 交易税 | 中 | 0% 或低 | 高税率（隐性成本） |

**2. 持仓分布**

| 检查项 | 权重 | 安全 | 危险 |
|--------|------|------|------|
| 最大单一持仓 | 高 | <5% | >10% |
| 前 10 持仓集中度 | 高 | <30% | >50% |
| 创建者持仓 | 高 | <2% | >5% |
| 关联地址检测 | 中 | 无关联 | 多个地址疑似同一人 |

**3. 创建者历史**

| 检查项 | 权重 | 安全 | 危险 |
|--------|------|------|------|
| 历史 Rug Pull 记录 | 极高 | 无 | 有 |
| 创建 Token 数量 | 中 | 少量 | 大量（批量发币） |
| 历史 Token 存活率 | 中 | 较高 | 极低 |

**4. 交易行为**

| 检查项 | 权重 | 安全 | 危险 |
|--------|------|------|------|
| 独立买家数量 | 中 | 多 | 少（可能是刷量） |
| 买卖比例 | 低 | 均衡 | 极度不均衡 |
| 大额卖出频率 | 中 | 低 | 高（内部人出货） |

### 评分算法

简化的评分逻辑：

```
总分 = 100

扣分规则：
  Mint 未撤销：-30
  Freeze 未撤销：-20
  LP 未锁定/销毁：-25
  单一地址持仓 >10%：-15
  前 10 集中度 >50%：-10
  创建者有 Rug 历史：-40
  疑似关联地址：-10
  交易税 >5%：-10

评级：
  80-100：🟢 低风险
  60-79：🟡 中风险
  40-59：🟠 高风险
  0-39：🔴 极高风险
```

### 技术实现

```
新 Token 被发现
    │
    ▼
快速检查（<1 秒，用于即时展示）
    ├── 检查 Mint/Freeze Authority
    ├── 检查 LP 状态
    └── 查询创建者黑名单
    │
    ▼
深度分析（1-10 秒，异步执行）
    ├── 分析持仓分布
    ├── 检测关联地址
    ├── 查询创建者历史
    └── 分析交易模式
    │
    ▼
持续监控（每分钟更新）
    ├── 持仓分布变化
    ├── 大额转账/卖出
    └── 异常交易模式
```

### 局限性

**1. 新型攻击手法**
- 安全检测基于已知的攻击模式
- 新的 Rug Pull 手法可能绕过现有检测
- 需要持续更新检测规则

**2. 误报和漏报**
- 过于严格 → 大量正常 Token 被标记为危险（误报）
- 过于宽松 → 危险 Token 没有被识别（漏报）
- 需要在两者之间找平衡

**3. 链上数据的局限**
- 有些风险无法从链上数据判断（如团队背景、项目真实性）
- 社交媒体信息难以自动化验证
- 跨链操作可能绕过检测

## 经验教训

1. **安全评分是概率判断，不是确定性结论** —— 高分不等于安全，低分不等于一定是骗局
2. **快速检查和深度分析要分开** —— 用户需要即时反馈，但深度分析需要时间
3. **持续监控比一次性检查更重要** —— Token 的风险状态会随时间变化
4. **透明展示检查结果** —— 让用户看到具体的检查项，而不只是一个分数
