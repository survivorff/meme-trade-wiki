# 交易为什么失败：Landing Rate 的真相与优化

> "交易失败"是 Meme 交易者最常遇到的问题。但大多数人不知道为什么失败，更不知道怎么优化。

## 表面认知

"交易失败就是网络太卡了。" 网络拥堵只是原因之一。

## 真实情况

### 什么是 Landing Rate

Landing Rate = 成功上链的交易数 / 提交的交易总数

一个好的交易平台的 Landing Rate 应该在 80-95%。低于 70% 说明有严重问题。

### 交易失败的常见原因

**1. 交易过期（Blockhash Expired）**
- 交易中的 Recent Blockhash 过期了（~1 分钟）
- 原因：交易构建后提交太慢，或网络拥堵导致确认延迟
- 解决：缩短从构建到提交的时间，使用更新的 Blockhash

**2. 滑点超限（Slippage Exceeded）**
- 实际价格变化超过了设置的滑点容忍度
- 原因：Token 价格波动太快，或流动性太差
- 解决：适当提高滑点设置，或减少交易金额

**3. Compute Unit 不足**
- 交易执行消耗的 CU 超过了设置的上限
- 原因：CU Limit 设置太低
- 解决：先模拟获取实际 CU，留足余量

**4. 余额不足**
- 账户余额不够支付交易金额 + 费用
- 原因：没有考虑 Priority Fee、Jito Tip 等额外费用
- 解决：在计算时预留足够的费用余量

**5. 网络拥堵**
- 区块空间不足，低优先级交易被丢弃
- 原因：热门 Token 发射等事件导致交易量暴增
- 解决：提高 Priority Fee，使用 Jito Bundle

**6. 流动性不足**
- DEX 上的流动性不够完成交易
- 原因：Token 太新或太冷门
- 解决：减少交易金额，或等待流动性改善

**7. 账户状态变化**
- 交易构建时的账户状态和执行时不一致
- 原因：在你的交易之前有其他交易改变了状态
- 解决：使用 Jito Bundle 保证执行环境

### 优化 Landing Rate 的策略

**平台侧优化：**

```
1. 交易模拟
   └── 提交前先模拟，过滤掉必然失败的交易

2. 动态参数调整
   ├── 根据网络状况动态调整 Priority Fee
   ├── 根据模拟结果精确设置 CU Limit
   └── 根据流动性动态调整滑点建议

3. 智能重试
   ├── 失败后自动分析原因
   ├── 根据原因调整参数重试
   ├── 设置最大重试次数和超时
   └── 避免重复提交（防止双花）

4. 多通道提交
   ├── 同时向多个 RPC 节点提交
   ├── 同时通过 Jito 和直接提交
   └── 哪个先成功用哪个

5. Blockhash 管理
   ├── 预缓存最新的 Blockhash
   ├── 定期刷新（每几秒）
   └── 确保交易使用最新的 Blockhash
```

### Landing Rate 的监控

平台需要实时监控：
- 整体 Landing Rate（按分钟/小时统计）
- 按失败原因分类的统计
- 按 Token/DEX 分类的统计
- 异常告警（Landing Rate 突然下降）

## 经验教训

1. **Landing Rate 是平台质量的核心指标** —— 直接影响用户体验和信任
2. **大多数失败是可以预防的** —— 通过模拟、动态参数、智能重试
3. **监控比优化更重要** —— 先知道问题在哪，才能针对性优化
4. **网络拥堵时要有降级策略** —— 不能让整个平台因为网络问题而瘫痪
