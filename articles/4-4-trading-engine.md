# 交易引擎：一笔买单如何变成链上交易

> 用户点击"买入"到链上确认，中间经历了十几个步骤。每一步都可能失败。

## 表面认知

"构建一笔交易发到链上就行了。" 如果只是一笔交易，确实不难。但要处理每秒数千笔交易，每笔都要快速、可靠、成本最优，那就是另一回事了。

## 真实情况

### 交易执行全流程

```
用户请求买入 1 SOL 的 Token X
    │
    ▼
① 参数校验
    ├── 用户余额是否足够？
    ├── Token 是否还在交易中？
    ├── 滑点设置是否合理？
    └── 是否触发风控规则？
    │
    ▼
② 路由选择
    ├── Token X 在哪些 DEX 上有流动性？
    ├── 各 DEX 的价格和深度如何？
    ├── 是否需要拆分到多个 DEX？
    └── 选择最优路由
    │
    ▼
③ 交易构建
    ├── 构建 Solana Transaction
    ├── 添加 DEX Swap 指令
    ├── 设置 Compute Unit Limit 和 Price
    ├── 设置 Priority Fee
    └── 如果使用 Jito：构建 Bundle
    │
    ▼
④ 交易模拟
    ├── 调用 simulateTransaction
    ├── 检查是否会成功
    ├── 获取预估的 Compute Unit 消耗
    └── 如果模拟失败 → 调整参数重试或返回错误
    │
    ▼
⑤ 签名
    ├── 托管钱包：后端用存储的私钥签名
    └── 非托管钱包：发回前端让用户钱包签名
    │
    ▼
⑥ 提交
    ├── Jito Bundle：提交到 Jito Block Engine
    └── 直接提交：发送到 RPC 节点
    │
    ▼
⑦ 状态追踪
    ├── 轮询交易状态（已确认/已最终化/失败）
    ├── 超时处理（多久算超时？超时后怎么办？）
    ├── 失败重试（是否自动重试？重试几次？）
    └── 成功确认 → 更新用户状态
```

### 路由选择的复杂性

同一个 Token 可能在多个 DEX 上有流动性：

```
Token X 的流动性分布：
  - PumpSwap: 60% 的流动性
  - Raydium: 30% 的流动性
  - Meteora: 10% 的流动性

买入 1 SOL 的最优策略可能是：
  - 0.6 SOL 走 PumpSwap
  - 0.3 SOL 走 Raydium
  - 0.1 SOL 走 Meteora

这样可以最小化滑点（每个池子的价格影响更小）
```

实际上大多数平台会使用 Jupiter 聚合器来处理路由，而不是自己实现：
- Jupiter 已经做了路由优化
- 支持多 DEX 拆单
- 但会增加一层依赖和延迟

### Compute Unit 管理

Solana 交易有 Compute Unit（CU）限制：

- 每笔交易默认 200,000 CU
- 复杂的 DEX Swap 可能需要 100,000-400,000 CU
- 设置太低 → 交易失败（CU 不足）
- 设置太高 → 浪费费用（按 CU 收费）

最佳实践：
1. 先模拟交易，获取实际 CU 消耗
2. 在实际消耗基础上加 20-30% 的余量
3. 根据网络状况动态调整 CU Price

### 失败处理

链上交易失败是常态，需要完善的失败处理：

| 失败原因 | 处理方式 |
|---------|---------|
| 滑点超限 | 提示用户调整滑点，或自动微调重试 |
| CU 不足 | 增加 CU Limit 重试 |
| 余额不足 | 提示用户 |
| 网络拥堵 | 增加 Priority Fee 重试 |
| 交易过期 | 用新的 Blockhash 重新构建 |
| 流动性不足 | 提示用户减少交易金额 |
| RPC 错误 | 切换到备用 RPC 重试 |

### 托管 vs 非托管的交易流程差异

| 维度 | 托管钱包 | 非托管钱包 |
|------|---------|-----------|
| 签名 | 后端直接签名 | 需要前端钱包签名 |
| 速度 | 更快（无需等待用户） | 较慢（需要用户确认） |
| 安全 | 平台持有私钥（风险） | 用户持有私钥（更安全） |
| 自动交易 | 可以实现 | 需要预授权或特殊机制 |
| 用户体验 | 更流畅 | 每笔交易需要确认 |

大多数 Meme 交易平台选择托管或半托管模式，因为速度在这个场景下至关重要。

## 经验教训

1. **交易模拟是必须的** —— 不模拟就提交，失败率会很高
2. **失败处理比成功路径更重要** —— 大部分工程量在处理各种异常
3. **监控每一步的延迟和成功率** —— 知道瓶颈在哪才能优化
4. **用户体验和安全性的平衡** —— 托管更快但风险更高
