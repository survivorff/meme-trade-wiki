# 行情数据管线：从原始区块到实时 K 线

> 用户看到的每一根 K 线、每一个价格跳动，背后是一条复杂的数据处理管线。

## 表面认知

"从链上读个价格有什么难的？" 难的不是读一个价格，而是为数百万个 Token 持续计算实时价格。

## 真实情况

### 价格从哪来

Meme 币没有中心化的报价系统。价格来自链上 DEX 的交易数据：

**Bonding Curve 阶段（Pump.fun 上）：**
- 价格由 Bonding Curve 数学公式决定
- 需要读取 Bonding Curve 合约的状态（当前供应量、储备金）
- 根据公式计算当前价格

**DEX 阶段（毕业后）：**
- 价格由流动性池中两种 Token 的比例决定
- 需要读取流动性池的状态
- 价格 = Token B 数量 / Token A 数量（简化）

**多 DEX 情况：**
- 同一个 Token 可能在多个 DEX 上有流动性池
- 需要聚合多个来源的价格
- 通常取流动性最深的池子的价格作为参考价

### 数据管线架构

```
链上原始数据
    │
    ▼
事件采集层
    ├── 监听 DEX 的 Swap 事件
    ├── 监听流动性池状态变化
    └── 监听 Bonding Curve 交易
    │
    ▼
数据解析层
    ├── 解析交易数据，提取价格和数量
    ├── 计算每笔交易的成交价
    └── 标记交易方向（买/卖）
    │
    ▼
聚合计算层
    ├── 计算实时价格（最新成交价）
    ├── 计算 OHLCV（开高低收量）K 线数据
    ├── 计算交易量统计
    └── 计算价格变化百分比
    │
    ▼
存储层
    ├── 实时数据 → Redis（热数据）
    ├── K 线数据 → TimescaleDB / ClickHouse
    └── 历史数据 → PostgreSQL
    │
    ▼
推送层
    ├── WebSocket 实时推送价格变化
    ├── REST API 提供历史数据查询
    └── 内部事件触发（止盈止损检查等）
```

### K 线计算

K 线（OHLCV）是交易者最常用的数据格式：

- **O**pen：时间段内第一笔交易的价格
- **H**igh：时间段内最高价
- **L**ow：时间段内最低价
- **C**lose：时间段内最后一笔交易的价格
- **V**olume：时间段内的总交易量

常见时间粒度：1 分钟、5 分钟、15 分钟、1 小时、4 小时、1 天

**计算挑战：**
- 需要为每个 Token 的每个时间粒度维护状态
- 活跃 Token 数量可能有数万个
- 每个新交易都需要更新对应的 K 线
- 需要处理跨时间段的边界情况

### 性能优化

**挑战数据：**
- 活跃 Token 数量：数万到数十万
- 每秒交易事件：数千到数万
- K 线时间粒度：6 种以上
- 并发 WebSocket 连接：数万

**优化策略：**

1. **分层缓存**
   - L1：内存中维护最近的价格和 K 线
   - L2：Redis 存储热门 Token 的完整 K 线
   - L3：数据库存储历史数据

2. **增量计算**
   - 不要每次都从头计算 K 线
   - 维护每个时间段的中间状态，新交易只更新增量

3. **按需计算**
   - 不是所有 Token 都需要实时 K 线
   - 只为有用户在看的 Token 计算实时数据
   - 其他 Token 延迟计算或按需计算

4. **批量推送**
   - 不要每笔交易都推送一次
   - 按固定间隔（如 100ms）批量推送更新

### 数据准确性问题

- **价格操纵**：小流动性池的价格容易被操纵
- **闪电贷攻击**：瞬间改变价格再恢复，K 线上会出现异常的影线
- **多池价差**：同一 Token 在不同 DEX 上价格可能不同
- **延迟差异**：不同数据源的延迟不同，可能导致短暂的价格不一致

## 经验教训

1. **行情数据是平台的基础** —— 数据不准，所有上层功能都不可靠
2. **性能优化是持续的工作** —— 随着 Token 数量增长，需要不断优化
3. **准确性和实时性需要平衡** —— 有时候稍微延迟但准确比实时但有误更好
4. **监控数据质量** —— 需要自动检测异常数据并告警
